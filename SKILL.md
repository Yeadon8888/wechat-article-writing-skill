# 公众号文章写作（系统化闭环版）

## 最重要原则：减少条目式输出

用流畅段落替代要点罗列。避免写成说明书式的 "1. 首先...2. 然后...3. 最后..."，改用逻辑递进的自然段落。检查清单、对比表格例外，但仍需克制。

---

## GitHub 同步工作流（必遵守）

这个 skill 目录现在以 GitHub 仓库为单一真相源，不再走本地多份副本互传。

每次开始处理前：

- 先在 skill 根目录执行 `git pull --rebase`
- 如果有冲突，先解决冲突，再继续修改

每次处理完成后：

- 执行 `git add -A`
- 用清楚的提交信息执行 `git commit -m "..."`
- 立即执行 `git push`

执行规则：

- 没有先 pull，不开始新一轮编辑
- 没有 push，不算本轮完成
- 所有内容系统文件、references、SKILL.md 变更都走同一仓库
- 提交信息优先使用 Conventional Commit，例如：`feat:`、`fix:`、`chore:`

## 一、创作工作流（从选题到发布）

完整闭环：**选题检索 → 素材检索 → 创作 → 发布复盘 → 沉淀方法论**

### 第零步：创作前检索（每次创作必做）

在开始写作前，先检索内容库，避免重复造轮子：

1. 在 `content-system/01-选题库/` 查找是否有已记录的相关选题
2. 在 `content-system/03-素材库/_索引.md` 检索可复用的框架、表达、案例
3. 找到后向用户展示，建议复用；找不到则直接进入第一步

### 第一步：加载资源

读取 `references/作者配置.md` 了解伊登的人设和写作风格。若作者配置缺失，先询问：笔名、领域、期望风格（亲和/严谨/幽默/干货）。根据文章类型，读取对应 references 文件。

### 第二步：搜索热点

用 WebSearch 搜索主题相关热点，找 2-3 个可切入角度。关注最新工具发布、行业动态、用户痛点讨论。

### 第三步：生成标题

参考 `references/写作方法.md` 中的 4 种标题公式，各生成 2-3 个选项，请用户选择。

### 第四步：搭建结构 → 创作内容

按用户选定的文章类型（工具介绍/方法教程/深度思考）使用对应结构模板。全程保持伊登的人设：凌晨时间感开头、情绪化词汇、短段落节奏、加粗强调关键观点。深度思考类文章额外读取 `references/深度写作.md`。

### 第五步：去AI化 + 质量检查

读取 `references/去AI化.md` 进行润色。对照爆款检查清单自查（标题冲突感、开头钩子、每 300 字一图、金句、互动引导）。

### 第六步：发布后复盘（发布后执行）

文章发布后，引导用户完成数据记录和方法论沉淀：

1. 将文稿移动到 `content-system/02-已发布/`
2. 在文稿末尾追加复盘块（读取 `references/内容系统.md` 中的复盘模板）
3. 根据数据表现判断是否沉淀到 `content-system/03-素材库/` 和 `content-system/04-方法论/`

## 二、内容系统管理

内容系统目录结构（项目根目录下）：

```
content-system/
├── 01-选题库/        # 待创作的选题
├── 02-已发布/        # 已发布文章 + 数据
├── 03-素材库/        # 可复用框架、表达、案例
└── 04-方法论/        # 从数据中提炼的创作规律
```

详细的检索规则、复盘模板、沉淀格式见 `references/内容系统.md`。
